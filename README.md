Here is a comprehensive architectural blueprint, system workflow document, and file-by-file inventory for the MINDSIGHT Assessment Platform. This documentation is explicitly structured to serve as an onboarding context package that you can hand directly to another AI developer to initialize cooperative coding, bug-fixing, or feature implementation.1. System Architecture & High-Level WorkflowMINDSIGHT is an end-to-end psychological profiling and clinical screening engine. It ingests multidimensional client answers (demographics, behavioral vectors, and mental health metrics) and synthesizes a unified, page-bounded evaluation report.The operational lifecycle is divided into three core execution layers:[ PHASE 1: OFFLINE TRAINING ]
Raw Datasets (CSV) ──> train_domainX.py ──> Binary Artifacts ──> models/saved_states/

[ PHASE 2: LIVE RUNTIME INFERENCE ]
Client Payload (responses.csv) ──> main.py ──> feature_mappings.py ──> inference_wrappers.py

[ PHASE 3: PROFILE AGGREGATION & REPORTING ]
Domain Metrics (Dict) ──> profile_aggregator.py ──> pdf_generator.py ──> Final PDF Report
Core Execution PhasesThe Offline Training Pipeline: Raw data vectors housed within datasets/ are processed by individual domain scripts (models/train_domain*.py). These scripts optimize psychometric or machine learning models, serializing the final model weights, thresholds, and feature registries directly into models/saved_states/.The Live Runtime Inference Pipeline: Raw consumer records are collected either via an interactive command-line interface (scripts/take_assessment.py) or via automated batch injection (tests/run_assessments.py). The pipeline triggers main.py, which routes the raw data through models/feature_mappings.py for cleaning and realignment before passing vectors into models/inference_wrappers.py for parallel evaluation.The Profile Aggregator & Reporting Pipeline: The inference wrapper outputs raw mathematical values and classification states to models/profile_aggregator.py. This engine standardizes the global response schema, which is finally consumed by models/pdf_generator.py to generate a clinical-grade PDF complete with metric gauges and local driver contribution profiles.2. Core Subsystems & Non-Trivial File DirectoryThe System Core & Orchestrationmain.py: The application controller. It manages the runtime initialization sequence, reads data payloads, coordinates feature alignment, calls the inference wrapper, and triggers report output generation.schema_config.json: The central structural data contract. It defines explicit value ranges, column indicators, baseline configuration settings, fallback parameters, and classification criteria for the entire architecture.Data Processing & Model Evaluation (models/)models/feature_mappings.py: The input normalization layer. It handles missing field lookups, data type coercions, and advanced feature engineering (such as parsing character time stamps like "23:30" or "07:00" into decimal hours to isolate net sleep intervals).models/inference_wrappers.py: The execution center of live inference. It loads serialized binaries into memory, passes vectorized items through specific algorithmic layers, and computes local feature attributions (SHAP values or probability-delta metrics).models/profile_aggregator.py: Synthesizes output dictionaries from all six evaluation modules, checks compliance against global response contracts, and formats categorical data into a clean JSON manifest.models/pdf_generator.py: A programmatic typesetting engine built on ReportLab. It translates the aggregated profile JSON into a polished PDF report featuring stylized severity layout grids, colored indicators, and local driver breakdown charts.The Domain Calibration & Training Enginesmodels/train_domain1_personality.py: Calibrates an Item Response Theory (IRT) model based on a multi-category Graded Response Model (GRM). It estimates item discrimination parameters ($\alpha$) and category boundary thresholds ($\beta$) across the Big Five core traits.models/train_domain2_self_esteem.py: Ingests Rosenberg scale variables, calculates raw scores, and derives empirical cumulative frequency matrices stratified across demographic blocks (age cohorts and gender).models/train_domain3_mood_sleep.py: Fits a multi-class LightGBM gradient-boosting classifier to group psychological resilience tiers based on joint sleep-wake characteristics and clinical indicators.models/train_domain4_multitask.py: Trains dual Random Forest regression architectures simultaneously to predict continuous scales for internet addiction and loneliness traits.models/train_domain5_burnout.py: Fits a continuous XGBoost regressor tracking occupational variables, establishing score risk tiers via optimized threshold boundaries.models/train_domain6_clinical.py: Generates a composite screening layer fitting an outlier-sensitive Isolation Forest anomaly model alongside a multi-class Logistic Regression classifier.3. Active Debugging & System Refinement RoadmapThe following technical anomalies have been verified against raw data records and require structural corrections across the code files. Share this checklist directly with the next AI developer:🟥 Priority 1: Domain 3 PHQ-9 Summation BoundariesProblem: The diagnostic sum under-represents clinical boundaries by an entire tier (e.g., displaying a score of 14 instead of 17).Root Cause Location: Look inside models/inference_wrappers.py and models/train_domain3_mood_sleep.py. The summation bounds are likely experiencing an off-by-one column range slice, or the functional impairment variable (DPQ100) is accidentally replacing one of the 9 primary diagnostic variables (DPQ010 through DPQ090).Fix: Ensure only variables DPQ010 to DPQ090 are included in the clinical classification sum.🟨 Priority 2: Domain 6 Driver Attribution DisconnectProblem: Individual driver contribution attributes return flat +0.0 outputs despite the client possessing multiple active, positive symptom indicators (such as overthinking or social avoidance).Root Cause Location: Check the explanation layer inside models/inference_wrappers.py. The module is rendering hardcoded fallback template descriptions instead of parsing the active feature vectors, or the probability-delta converter is zeroing out.Fix: Ensure that driver calculations map directly to the non-zero symptoms present in the user payload and return correct impact margins scaled to probability space.🟨 Priority 3: Domain 2 Valence Inversion & Scoring DiscrepancyProblem: Raw Rosenberg Self-Esteem (RSE) sums do not match standard scoring rules, and the drivers show unsigned values (+4) for items that should lower self-esteem.Root Cause Location: Review the item index mapping arrays inside models/feature_mappings.py and models/inference_wrappers.py. Positively and negatively phrased scale elements are likely being added uniformly without applying the standard clinical inversion ($5 - \text{score}$) to the negative elements (Q2, Q6, Q8, Q9).Fix: Apply the score inversion to negative items and ensure the driver signs correctly indicate whether they raise ($+$) or lower ($-$) overall self-esteem.🟦 Priority 4: Domain 1 Latent Trait DroppingProblem: The final report displays local drivers for traits like Conscientiousness, but the core placement metrics drop Conscientiousness (CSN) and Openness (OPN) entirely from the results.Root Cause Location: Check models/pdf_generator.py and models/profile_aggregator.py. The GRM calibration data tracks all 5 dimensions, but the rendering layout logic is capped or hardcoded to display only the first 3 traits (EXT, EST, AGR).Fix: Expand the PDF rendering loop to handle all five trait variables.🟦 Priority 5: Domain 4 Undifferentiated Driver StreamsProblem: Feature contributions for both the Internet Addiction Test (IAT) and the UCLA Loneliness scale are dumped into a single list, making it impossible to see which driver explains which model score.Root Cause Location: Look at the dictionary composition in models/inference_wrappers.py and the layout block in models/pdf_generator.py.Fix: Split the single attribution array into two labeled sub-lists within the final layout schema.
